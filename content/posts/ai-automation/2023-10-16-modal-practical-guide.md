`Modal`은 2026년 기준으로 `serverless AI infra`, `Modal`, `inference`, `batch jobs`, `AI sandboxes` 같은 검색어에서 꾸준히 언급되는 주제입니다. 이유는 단순합니다. AI 제품은 모델 호출만으로 끝나지 않고, 배치, 추론, 트레이닝, 실험, 코드 실행 샌드박스, GPU 스케일링까지 한 묶음으로 운영해야 하기 때문입니다.

Modal 공식 문서는 inference, sandboxes, batch processing, training을 각각 분리해서 다룹니다. 특히 Sandboxes 문서에서는 `execute code generated by a language model`, `isolated environments`, `check out a git repository and run a command against it` 같은 용례를 직접 제시합니다. 즉 `Modal이란`, `serverless AI infra`, `code-defined infra`, `AI sandbox` 같은 검색 의도와 잘 맞습니다.

![Modal 워크플로우](/images/modal-workflow-2026.svg)

## 이런 분께 추천합니다

- AI 추론과 배치를 서버리스 형태로 운영하고 싶은 팀
- 안전한 코드 실행 샌드박스가 필요한 개발자
- `Modal`, `AI sandbox`, `serverless inference`를 비교 중인 분

## Modal의 핵심은 무엇인가

핵심은 "Python 코드로 인프라와 실행 환경을 정의하고, 추론과 배치를 같은 플랫폼에서 다룬다"는 점입니다.

| 기능 | 의미 |
|---|---|
| Inference | 모델 서빙과 스케일링 |
| Batch | 대량 작업과 백그라운드 처리 |
| Training | 모델 학습과 반복 실험 |
| Sandboxes | 격리된 코드 실행 |
| Code-defined infra | 인프라를 코드로 선언 |
| GPU access | 최신 GPU 사용과 확장 |

즉 Modal은 단순한 job runner가 아니라, AI 실행 인프라의 여러 층을 한 도구에 넣은 형태입니다.

## 왜 지금 Modal이 중요한가

AI 운영에서는 다음 문제가 자주 반복됩니다.

- 추론은 빠른데 배치가 따로 놀음
- 샌드박스가 없어서 에이전트 코드 실행이 위험함
- 학습과 서빙 환경이 갈라져 운영이 복잡함
- GPU가 필요할 때마다 인프라를 다시 설계해야 함

Modal은 이런 문제를 `code-first` 접근으로 줄입니다. 그래서 `Modal inference`, `Modal sandboxes`, `serverless GPU` 같은 검색 유입을 잡기 좋습니다.

## 어떤 팀에 잘 맞는가

- Python 중심 AI 팀
- 에이전트용 코드 실행 환경이 필요한 팀
- batch inference와 training을 함께 운영하는 팀
- 인프라보다는 제품 로직에 집중하고 싶은 스타트업

## 실무 도입 시 체크할 점

1. 추론, 배치, 트레이닝, 샌드박스를 서로 다른 경계로 나눕니다.
2. 샌드박스에는 timeout과 라이프사이클 정책을 명확히 둡니다.
3. GPU 자원과 cold start 요구를 먼저 점검합니다.
4. 코드 정의와 운영 설정을 분리합니다.
5. 로그와 비용 관측을 초반부터 붙입니다.

## 장점과 주의점

장점:

- Python 코드 중심이라 빠르게 시작할 수 있습니다.
- inference와 batch를 같은 관점에서 다룰 수 있습니다.
- sandbox 문서가 에이전트 코드 실행에 직접 유용합니다.
- GPU 인프라 운영 복잡도를 줄이기 좋습니다.

주의점:

- 코드로 인프라를 정의해도 운영 정책은 별도로 필요합니다.
- 배치, 서빙, 샌드박스의 비용 구조가 다릅니다.
- 앱 경계가 커지면 로그와 리소스 분리가 중요해집니다.
